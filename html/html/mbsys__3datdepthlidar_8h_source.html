33 /*
34  * Notes on the MBSYS_3DATDEPTHLIDAR data structure:
35  *
36  * Vendor processed format from 3DatDepth, produced from raw LIDAR
37  * time series files by proprietary 3DatDepth software
38  *
39  *--------------------------------------------------------------------------------
40  * Processing Tool Data Output
41  *
42  * The 3D at Depth Processing Tool reads binary sensor data and
43  * processes it to provide corresponding range angle data, saved
44  * to a binary file. A timestamp is provided for each scan recorded
45  * in the file. This timestamp is the time of the first laser pulse
46  * for that scan. The μsec time value listed for each laser pulse
47  * is the relative time between each successive pulse, offset from
48  * the timestamp. The value for the first pulse should be subtracted
49  * from all pulses in that scan (including itself), such that the
50  * first pulse would be 0 μsec accordingly.
51  *
52  * ---------------------------------------------------------------------------------------
53  * Range Angle Angle data format (binary)
54  * Item Value Bytes
55  * ---------------------------------------------------------------------------------------
56  * File Header Record
57  * File magic number 0x3D46 2 (1 UINT16)
58  * Parameter record id 0x3D07 2 (1 UINT16)
59  * File version 1 2 (1 UINT16)
60  * File sub version 1 2 (1 UINT16)
61  * Scan Information
62  * Scan type (AZ raster, AZEL raster, bowtie) 2, 3, 4 2 (1 UINT16)
63  * Cross track angle start (deg) 4 (1 float32)
64  * Cross track angle end (deg) 4 (1 float32)
65  * Forward track angle start (deg) 4 (1 float32)
66  * Forward track angle end (deg) 4 (1 float32)
67  * Counts per Scan (AZ raster and bowtie) 2 (1 UINT16)
68  * Counts per cross track (AZEL raster) 2 (1 UINT16)
69  * Counts per forward track (AZEL raster) 2 (1 UINT16)
70  * Scanner Efficiency 2 (1 UINT16)
71  * Scans per File 2 (1 UINT16)
72  * Scan count 4 (1 UINT32)
73  *
74  * ---------------------------------------------------------------------------------------
75  * Lidar Scan Record
76  * Lidar scan record id 0x3D52 2 (1 UINT16)
77  * First Pulse Timestamp ( 1 to n Scans )
78  * Timestamp year 2 (1 UINT16)
79  * Timestamp month 1 (1 UINT8)
80  * Timestamp day 1 (1 UINT8)
81  * Timestamp days since Jan 1 2 (1 UINT16)
82  * Timestamp hour 2 (1 UINT16)
83  * Timestamp minutes 1 (1 UINT8)
84  * Timestamp seconds 1 (1 UINT8)
85  * Timestamp nano seconds 4 (1 UINT32)
86  * Laser Pulse Data ( 1 to m pulses per scan )
87  * Range ( from glass front ) meters 4 (1 float32)
88  * Amplitude / peak of signal 2 (1 short int)
89  * SNR of signal return 4 (1 float32)
90  * Cross track angle (deg) 4 (1 float32)
91  * Forward track angle (deg) 4 (1 float32)
92  * Cross track offset (m) 4 (1 float32)
93  * Forward track offset (m) 4 (1 float32)
94  * Pulse time offset (µsec) 4 (1 UINT32)
95  * Saturated (0/1) 1 (1 UINT8)
96  *
97  * For each scan per file, a “First Pulse Timestamp” and “m” sets of
98  * “Laser Pulse Data” will exist. For example, for a Continuous Scan
99  * file with 500 scans per file, and 200 pts per scan, the following
100  * data would be present:
101  * File Header
102  * Scan Information
103  * (1) First Pulse Timestamp
104  * 200 Laser Pulse Data sets
105  * (2) First Pulse Timestamp
106  * 200 Laser Pulse Data sets
107  * …
108  * (500) First Pulse Timestamp
109  * 200 Laser Pulse Data sets
110  *
111  * A Bowtie scan file would be the same as above. A Full scan file
112  * will only contain one scan of data but with “Counts per Cross track”
113  * multiplied by “Counts per Forward track” sets of laser pulse data.
114  *
115  * The timestamp is the time of the first laser pulse for that scan.
116  * The pulse time offset (µsec) value listed for each laser pulse is
117  * the relative time between each successive pulse. The value for the
118  * first pulse should be subtracted from all pulses in that scan, such
119  * that the first pulse would be 0 µsec accordingly. The Pt Valid term
120  * is provided to specify range validity. If “valid” is given as 0, a
121  * range could not be determined for that laser pulse, or it fell
122  * outside of the processing limits.
123  *
124  * Note, if processing is interrupted by hitting the “Cancel Processing”
125  * button, an indeterminate number of records may exist in the current
126  * processing file. All processed files will be written to the same
127  * folder location of the raw file. The same file name is used for
128  * processed files, but with a “.csv” or “.bin” file extension. If data
129  * is reprocessed in the same folder location, previously processed data
130  * files will be overwritten.
131  *
132  *--------------------------------------------------------------------------------
133  * INS Data Format
134  *
135  * INU serial packets are archived to data files located at %RAW_DATA_PATH%/INU/.
136  * %RAW_DATA_PATH% is set in the sensor configuration file. Data files are
137  * named INU_MMDDYYYY_HHMMSS.bin. 3000 navigation messages are stored per
138  * each archived data file, representing approximately 5 minutes data storage
139  * if collecting messages at 10 Hz. The binary data file storage is provided
140  * below, for the first record. Additional records follow to 3000 per file.
141  * The latency from first character received to the timestamp is approximately
142  * 100 msec. If an error occurs, or the first byte read from the serial port is
143  * not equal to the INU magic character, the software will begin reading byte
144  * by byte in an attempt to resync.
145  * Item Value Bytes
146  * Header Magic Number 0xE32F 2 (1 UINT16)
147  * Timestamp year 2 (1 UINT16)
148  * Timestamp month 1 (1 UINT8)
149  * Timestamp day 1 (1 UINT8)
150  * Timestamp days since Jan 1 2 (1 UINT16)
151  * Timestamp hour 2 (1 UINT16)
152  * Timestamp minutes 1 (1 UINT8)
153  * Timestamp seconds 1 (1 UINT8)
154  * Timestamp nano seconds 4 (1 UINT32)
155  * Latency (μseconds) – latency from first 4 (1 float32)
156  * character received to timestamp
157  * INU Nav message 1
158  * complete message as received from INU Len / packet ‐ reference the INU specification
159  * 1 magic, 1 ID, 1 data len, N NAV bytes
160  *
161  *--------------------------------------------------------------------------------
162  * CTD Ethernet Interface
163  *
164  * The 3D at Depth laser sensor listens for CTD data packets available on port
165  * 2003 of the static IP address for the sensor. The sensor reads and timestamps
166  * these packets and saves the data to a binary file, located at %RAW_DATA_PATH%/CTD/.
167  * %RAW_DATA_PATH% is set in the sensor configuration file.
168  * Data files are named CTD_MMDDYYYY_HHMMSS.bin. The file format is as follows.
169  * CTD Packets
170  * Item Value Bytes
171  * Header Magic Number 0x3D07 2 (1 UINT16)
172  * Timestamp year 2 (1 UINT16)
173  * Timestamp month 1 (1 UINT8)
174  * Timestamp day 1 (1 UINT8)
175  * Timestamp days since Jan 1 2 (1 UINT16)
176  * Timestamp hour 2 (1 UINT16)
177  * Timestamp minutes 1 (1 UINT8)
178  * Timestamp seconds 1 (1 UINT8)
179  * Timestamp nano seconds 4 (1 UINT32)
180  * Header project code 0x3D03 2 (1 UINT16)
181  * Header version 1 1 (1 UINT8)
182  * Water Temperature C 8 (1 double)
183  * Water Salinity psu 8 (1 double)
184  * Water Pressure dbar 8 (1 double)
185  *--------------------------------------------------------------------------------
186  *
187  */
